# <a name="iwiatransfercallbackgetnextstream-method"></a>Метод Ивиатрансферкаллбакк:: Жетнекстстреам

Возвращает новый поток для указанного элемента.

## <a name="syntax"></a>Синтаксис


```C++
HRESULT GetNextStream(
  [in]  LONG    lFlags,
  [in]  BSTR    bstrItemName,
  [in]  BSTR    bstrFullItemName,
  [out] IStream **ppDestination
);
```



## <a name="parameters"></a>Параметры

<dl> <dt>

*лфлагс* \[ окне\]
</dt> <dd>

Тип: **Long**

В настоящее время не используется. Значение должно быть равно нулю.

</dd> <dt>

*бстритемнаме* \[ окне\]
</dt> <dd>

Тип: **BSTR**

Указывает имя элемента, для которого создается поток.

</dd> <dt>

*бстрфуллитемнаме* \[ окне\]
</dt> <dd>

Тип: **BSTR**

Указывает полное имя элемента, для которого создается поток.

</dd> <dt>

*ппдестинатион* \[ заполняет\]
</dt> <dd>

Тип: **[IStream](/windows/win32/api/objidl/nn-objidl-istream)\*\***

Получает адрес указателя на новый объект [IStream](/windows/win32/api/objidl/nn-objidl-istream) .

</dd> </dl>

## <a name="return-value"></a>Возвращаемое значение

Тип: **HRESULT**

Если этот метод завершается успешно, возвращается значение **S \_ ОК**. В противном случае возвращается код ошибки **HRESULT** .

## <a name="remarks"></a>Комментарии

Если этот метод реализуется с помощью фильтра обработки изображений, то при получении образа Windows (WIA) 2,0 минидривер вызывает его во время получения изображения, чтобы получить целевой поток от клиента.

Фильтр **ивиатрансферкаллбакк:: жетнекстстреам** должен делегировать методу обратного вызова приложения. Фильтр использует поток, возвращенный реализацией **ивиатрансферкаллбакк:: жетнекстстреам** обратного вызова приложения, чтобы создать собственный поток, который передается обратно в службу WIA 2,0. Фильтрация выполняется, когда поток фильтра вызывает метод [IStream:: Write](/windows/win32/api/objidl/nf-objidl-isequentialstream-write) .

Поток фильтра не может делать никаких предположений о количестве байтов, записываемых в него при каждой записи, поскольку нефильтрованные данные изображения могут поступать из компонента предварительной версии WIA 2,0, а не драйвера. Компонент WIA 2,0 Preview всегда записывает все нефильтрованные данные изображения в поток фильтра только один раз, что означает, что поток фильтра имеет один источник. Если драйвер и компонент предварительного просмотра записывают данные в поток фильтра, поток фильтра не может предположить, например, что он получит полный заголовок при первом вызове [IStream:: Write](/windows/win32/api/objidl/nf-objidl-isequentialstream-write) , несмотря на то, что соответствующий драйвер всегда сначала записывает данных заголовка в одну запись. Также нельзя предположить, что последующая запись содержит только одну строку сканирования. Поэтому потоку фильтрации может потребоваться подсчитать число байтов, записанных в него, чтобы определить, например, начало данных изображения.

Реализация **ивиатрансферкаллбакк:: жетнекстстреам** фильтра обработки изображений должна считывать свойства, необходимые для обработки изображения из элемента, для которого запрашивается изображение. Фильтр не считывает свойства непосредственно из *pWiaItem2* , переданного в [**инитиализефилтер**](-wia-iwiaimagefilter-initializefilter.md). Вместо этого фильтр должен вызвать [**финдитембинаме**](-wia-iwiaitem2-finditembyname.md) для этого элемента WIA 2,0, чтобы получить фактический элемент WIA 2,0. Причина в том, что получаемый образ может фактически быть дочерним элементом *pWiaItem2*. Например, во время приобретения папки фильтр использует *pWiaItem2* для получения дочерних элементов *pWiaItem2* в **ивиатрансферкаллбакк:: жетнекстстреам** (во время приобретения папки драйвер возвращает образы, представленные дочерними элементами *pWiaItem2*). То же самое верно, когда компонент WIA 2,0 Preview вызывает фильтр обработки изображений, передающий дочерний элемент WIA 2,0.
