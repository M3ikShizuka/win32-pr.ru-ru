# <a name="ntlmssp"></a>NTLMSSP

Поставщик NTLMSSP, чей идентификатор службы проверки подлинности — RPC \_ C \_ AUTHN \_ WinNT, является поставщиком поддержки безопасности, доступным во всех версиях DCOM. Для проверки подлинности используется протокол NTLM. NTLM никогда не передает пароль пользователя на сервер во время проверки подлинности. Поэтому сервер не может использовать пароль во время олицетворения для доступа к сетевым ресурсам, к которым у пользователя есть доступ. Доступны только локальные ресурсы.

NTLM работает как локально, так и на нескольких компьютерах. То есть, если клиент и сервер находятся на разных компьютерах, NTLM по-прежнему может убедиться, что клиент является им.

При использовании NTLM удостоверение клиента представляется именем домена, именем пользователя, паролем или токеном. Когда сервер вызывает [**кокуериклиентбланкет**](/windows/desktop/api/combaseapi/nf-combaseapi-coqueryclientblanket), возвращаются доменное имя клиента и имя пользователя. Однако когда сервер вызывает [**коимперсонатеклиент**](/windows/desktop/api/combaseapi/nf-combaseapi-coimpersonateclient), возвращается маркер клиента. Если между клиентом и сервером нет отношений доверия и если у сервера есть локальная учетная запись с тем же именем и паролем, что и у клиента, то эта учетная запись будет использоваться для представления клиента.

NTLM поддерживает взаимную проверку подлинности между потоками и между процессами (только локально). Если клиент указывает имя участника сервера в форме "пользователь домена" \\ при вызове [**иклиентсекурити:: сетбланкет**](/windows/win32/api/objidl/nf-objidl-iclientsecurity-setblanket), удостоверение сервера должно совпадать с именем участника, иначе вызов завершится ошибкой. Если клиент указывает **значение NULL**, удостоверение сервера не будет проверяться.

NTLM дополнительно поддерживает уровень олицетворения делегирования между потоками и между процессами (только локально).
