# <a name="a-isolated-applications-and-side-by-side-assemblies"></a>A (изолированные приложения и параллельные сборки)

A B C [D](d-sbscs-gly.md) E F [G](g-sbscs-gly.md) р [. J K](i-sbscs-gly.md) L [M](m-sbscs-gly.md) N O [P](p-sbscs-gly.md) Q [R](r-sbscs-gly.md) [S](s-sbscs-gly.md) T U V W X Y Z

<dl> <dt>

<span id="_win32_activation_context_gly"></span><span id="_WIN32_ACTIVATION_CONTEXT_GLY"></span>**контекст активации**
</dt> <dd>

Структура данных в памяти. Каждый раздел этой структуры содержит метаданные для параллельных функций API. Например, один раздел может иметь данные перенаправления DLL, которые используются загрузчиком DLL, а другая может иметь данные COM-сервера. Эти данные могут использоваться для перенаправления загрузки библиотеки DLL в определенную версию, создания COM-объекта или создания окна до версии, которая является наиболее совместимой для приложения.

</dd> <dt>

<span id="_win32_application_configuration_gly"></span><span id="_WIN32_APPLICATION_CONFIGURATION_GLY"></span>**Конфигурация приложения**
</dt> <dd>

Имена и версии параллельных сборок, необходимых для запуска приложения. При развертывании приложения с манифестом явно определяются зависимости от конкретных версий общих параллельных сборок. По умолчанию версия сборки, указанная в манифесте, является версией, используемой при активации. Глобальная конфигурация приложения задает конфигурацию всех приложений в системе. Конфигурация для каждого приложения может переопределять глобальную конфигурацию приложения на основе каждого приложения.

</dd> <dt>

<span id="_win32_application_configuration_manifest_gly"></span><span id="_WIN32_APPLICATION_CONFIGURATION_MANIFEST_GLY"></span>**Манифест конфигурации приложения**
</dt> <dd>

Файл, указывающий параллельные сборки, которые будут использоваться полностью или частично изолированным приложением. Файлы манифеста конфигурации приложения устанавливаются в ту же папку, что и исполняемый файл приложения.

</dd> <dt>

<span id="_win32_application_manifest_gly"></span><span id="_WIN32_APPLICATION_MANIFEST_GLY"></span>**манифест приложения**
</dt> <dd>

Файл, описывающий изолированное приложение. Он задает сведения, необходимые для запуска приложения, включая зависимости от закрытых сборок, конкретные версии общих сборок и метаданных для закрытых сборок. Имя файла манифеста приложения — это имя исполняемого объекта приложения, за которым следует расширение MANIFEST. Например, для MySampleApp.exe файл манифеста будет MySampleApp.exe. manifest.

</dd> <dt>

<span id="_win32_assembly_gly"></span><span id="_WIN32_ASSEMBLY_GLY"></span>**сборок**
</dt> <dd>

Фундаментальный модуль для именования, привязки, управления версиями, развертывания или настройки блока программного кода. Эти сборки кода могут размещаться в DLL-библиотеках или сборках COM. Приложения с общей функциональностью могут выполнять общие блоки программного кода, которые называются модулями или сборками кода. Инфраструктура безопасного совместного использования сборок называется параллельным совместным доступом к сборкам.

</dd> <dt>

<span id="_win32_assembly_manifest_gly"></span><span id="_WIN32_ASSEMBLY_MANIFEST_GLY"></span>**манифест сборки**
</dt> <dd>

Описание параллельной сборки. Он задает имя, версию, файлы, ресурсы сборки, привязку данных для элементов сборки и зависимости от других параллельных сборок. Файл манифеста сборки может иметь любое допустимое имя файла, если за ним следует расширение MANIFEST.

</dd> </dl>
