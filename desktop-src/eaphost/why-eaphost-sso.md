# <a name="sso-eaphost-scenario-overview"></a>Обзор сценария с единым входом EAPHost

В следующем разделе содержатся два сценария, демонстрирующие преимущества одноразовых запросов с поддержкой единого входа.

## <a name="about-the-scenarios"></a>О сценариях

Единый вход предоставляет функциональные возможности для хранения дополнительных сведений об учетных данных пользователя, чем обычно хранится в пользовательском BLOB-объекте EAP. В отличие от других процессов учетных данных, отправителей запросов с поддержкой единого входа может разрешать такие ситуации, как перемещение AP, и изменение пароля без вмешательства пользователя.

## <a name="sso-eap-tls-pin-caching-behavior"></a>Поведение кэширования для единого входа EAP-TLS

Запрашивающая сторона может попытаться выполнить проверку подлинности сети с помощью метода EAP на основе смарт-карты, такого как протокол EAP-TLS. В этом и подобных сценариях запрашивающая сторона получает от пользователя ПИН-код и получает сертификат пользователя для проверки подлинности сети, за которым следует вызов Винлогин на локальном компьютере. После успешной проверки подлинности сторона кэширует большой двоичный объект пользователя и не сохраняет информацию о ПИН-коде пользователя.

Когда пользователь переходит в другое место, необходимо возобновить сеанс и повторно пройти проверку подлинности. Так как сертификат не может быть сохранен при предварительном входе в систему, проверка подлинности пользователя завершится ошибкой, а запрашивающий объект запустит Пользовательский BLOB. На этом этапе для получения сертификата пользователя из смарт-карты для сетевой проверки подлинности требуется ПИН пользователя. Поскольку единый вход кэширует ПИН-код, сертификат может быть получен без вмешательства пользователя. Без единого входа EAP-TLS должен снова вызвать пользовательский интерфейс коллекции закрепления.

Пошаговый подход см. в разделе [режим кэширования ПИН-кода в режиме единого входа](sso-eap-tls-pin-caching-behavior-.md).

## <a name="sso-password-change-behavior"></a>Поведение при изменении пароля единого входа

Запрашивающий объект может попытаться выполнить проверку подлинности в сети с помощью метода EAP на основе пароля, например протокола проверки подлинности Microsoft Challenge версии 2,0 (MS-CHAPv2), настроенного для использования учетных данных Винлогин. В этом сценарии запрашивающая сторона собирает учетные данные пользователя один раз и предоставляет им значение EAPHost для проверки подлинности сети. После успешной проверки подлинности сети запрашивающий объект использует тот же набор учетных данных для Винлогин.

Однако, если учетные данные, такие как пароль пользователя, были изменены во время проверки подлинности сети без запрашивающего абонента, последующий вызов Винлогин приведет к сбою. Единый вход оставляет новые учетные данные и позволяет успешно Винлогин без дополнительного вмешательства пользователя.

Пошаговый подход см. в разделе поведение при [изменении пароля единого входа](sso-password-change-behavior-.md).
