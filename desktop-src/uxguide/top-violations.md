# <a name="ux-checklist-for-desktop-applications"></a>Контрольный список UX для настольных приложений

> [!NOTE]
> Это руководство по проектированию было создано для Windows 7 и не Обновлено для более новых версий Windows. Многие рекомендации по-прежнему применяются в принципе, но презентация и примеры не соответствуют нашим [текущим руководствам по проектированию](/windows/uwp/design/).

Ниже приведена часть важных рекомендаций в руководстве по UX. Вы можете использовать этот контрольный список, чтобы убедиться, что пользовательский интерфейс программы получает эти важные элементы правее.

## <a name="windows"></a>Windows

-   **Поддерживают минимальное эффективное разрешение Windows (800x600 пикселей).** Для критически важных пользовательских интерфейсов (UI), которые должны работать в защищенном режиме, поддерживают [эффективное разрешение](glossary.md) 640 пикселей. Заследите за пространством, используемым панелью задач, выполнив резервирование 48 [относительных](glossary.md) вертикальных пикселей для окон, отображаемых на панели задач.
-   **Оптимизируйте макеты окон с изменяемыми размерами для эффективного разрешения 1024x768 пикселей.** Автоматическое изменение размера этих окон для более низких разрешений экрана.
-   **Обязательно протестируйте Windows в 96 точках на дюйм (DPI) (с разрешением 800x600 пикселей), 120 dpi (на 1024x768 пикселей) и 144 dpi (в 1200x900 пикселях).** Проверьте наличие проблем макета, таких как обрезка элементов управления, текст и окна, а также растяжение значков и растровых изображений.
-   **Для программ с сенсорным и мобильным вариантами использования Оптимизируйте для 120 dpi.** Экраны с высоким разрешением в настоящее время являются наиболее распространенными на сенсорных и мобильных ПК.
-   **Если окно является владельцем окна, оно изначально отображается в центре окна, расположенного на экране владельца.** Никогда не ниже. Для последующего отображения рассмотрите возможность отображения его в последнем расположении (относительно окна "владелец"), если это может быть более удобным.
-   **Если окно является контекстным, всегда отображать его рядом с объектом, из которого он был запущен.** Тем не менее, разместите его вне пути (желательно сместить вниз и вправо), чтобы объект не покрыт окном.

## <a name="layout"></a>Layout

-   **Размеры элементов управления и панелей в окне в соответствии с их типичным содержимым.** Избегайте усечения текста и связанных с ними эллипсов. Пользователям никогда не нужно взаимодействовать с окном для просмотра его обычного содержимого — резервного изменения размера и прокрутки для необычно большого содержимого. Конкретная проверка:
    -   **Размеры элементов управления.** Размер элементов управления в обычном содержимом, что делает элементы управления шире, более высокой или многострочной при необходимости. Размер элементов управления для исключения или уменьшения прокрутки в Windows, имеющих много свободного места. Кроме того, в Windows не должно быть усеченных меток или обрезанного текста, который содержит достаточно свободного места. Однако, чтобы сделать текст более удобным для чтения, рассмотрите возможность ограничения ширины линии до 65 символов.
    -   **Ширина столбцов.** Убедитесь, что столбцы представления списка имеют подходящие значения по умолчанию, минимальный и максимальный размер. Выберите список представления ширина столбцов по умолчанию, которые не приводят к усеченному тексту, особенно если в представлении списка имеется свободное место.
    -   **Баланс макета.** Макет окна должен быть приблизительно сбалансированным. Если макет выглядит по-левому краю, рекомендуется сделать элементы управления шире и переместить некоторые элементы управления вправо.
    -   **Изменение размера макета.** Если размеры окна изменяются и данные усекаются, убедитесь, что для большего размера окна отображается больше данных. При усечении данных пользователи предполагают изменение размера окон для отображения дополнительных сведений.
-   **Установите минимальный размер окна, если ниже приведен размер содержимого, который больше не может использоваться.** Для элементов управления с изменяемыми размерами Задайте минимальный размер элементов для минимального размера, например минимальную ширину функциональных столбцов в представлениях списка.

## <a name="text"></a>Текст

-   **При возможности используйте обычные условия для общения.** Сосредоточьтесь на пользовательских целях, а не на технологиях. Это особенно эффективно, если вы объясняем сложную техническую концепцию или действие. Представьте себе, что вы просматриваете свое право пользователя и объясняете, как выполнить задачу.
-   **Будьте в повежливо, поддержке и поощрении.** Пользователь никогда не должен иметь права на изменен или пугайтесь.
-   **Удалите избыточный текст.** Поиск избыточного текста в заголовках окон, основных инструкциях, дополнительных инструкциях, областях содержимого, ссылках на команды и кнопках фиксации. Как правило, следует оставлять полный текст в основных инструкциях и интерактивных элементах управления, а также удалять все избыточность из других мест.
-   **Используйте прописные буквы в стиле заголовка для заголовков и капитализации в стиле предложений для всех других элементов пользовательского интерфейса.** Это более подходит для тона Windows.
    -   **Исключение:** Для устаревших приложений можно использовать прописные буквы в стиле заголовка для кнопок, меню и заголовков столбцов, если это необходимо, чтобы избежать смешивания стилей капитализации.
-   **Для имен компонентов и технологий будьте осторожны при капитализации.** Как правило, только основные компоненты должны быть прописными буквами (с использованием капитализации в стиле заголовка).
-   **Имена компонентов и технологий должны быть одинаковыми при капитализации.** Если имя встречается несколько раз на экране пользовательского интерфейса, оно должно всегда отображаться одинаковым образом. Аналогично, на всех экранах пользовательского интерфейса в программе имя должно быть постоянно представлено.
-   **Не используйте прописные буквы в именах общих элементов пользовательского интерфейса,** таких как панель инструментов, меню, полоса прокрутки, кнопка и значок.
    -   **Исключения:** Адресная строка, панель ссылок, лента.
-   **Не используйте все прописные буквы для клавиш клавиатуры.** Вместо этого следует следовать капитализации, используемой стандартными клавиатурами, или строчными буквами, если ключ не помечен на клавиатуре.
-   **Многоточие означает неполноту.** Используйте многоточие в тексте пользовательского интерфейса следующим образом:
    -   **Меню.** Укажите, что команде требуются дополнительные сведения. Не используйте многоточие, когда действие отображает другое окно — только в том случае, если требуется дополнительная информация. Команды, неявные команды которых покажут другое окно, не принимают многоточие, например дополнительные параметры, справку, настройки, свойства или параметры.
    -   **Данные.** Указывает, что текст усекается.
    -   **Сигнатур.** Укажите, что задача выполняется (например, "Поиск...").

**Совет.** Обрезанный текст в окне или странице с неиспользуемым пространством указывает на плохой макет или размер окна по умолчанию слишком мал. Стремитесь к макетам и размерам окон по умолчанию, которые устраняют или уменьшают объем усеченного текста. Дополнительные сведения см. в разделе [Макет](vis-layout.md).

-   **Не используйте синий текст, который не является ссылкой, так как пользователи могут предположить, что это ссылка.** Используйте полужирный или оттенок серого, где в противном случае используется цветной текст.
-   **Используйте полужирное начертание, чтобы привлечь внимание к текстовым пользователям, которые должны читаться.**
-   **Используйте основную инструкцию, чтобы объяснить, что пользователи должны делать в определенном окне или странице.** Хорошие основные инструкции сообщают о цели пользователя, а не на том, чтобы сосредоточиться только на управлении пользовательским интерфейсом.
-   **Выполните главную инструкцию в форме императивного направления или конкретного вопроса.**
-   **Не размещайте точки в конце меток управления или основных инструкций.**
-   **Используйте один пробел между предложениями.** Не два.

## <a name="controls"></a>Элементы управления

-   **Общие сведения**
    -   **Пометка для каждого элемента управления или группы элементов управления. Отличи**
        -   Текстовые поля и раскрывающиеся списки можно помечать с помощью запросов.
        -   Подчиненные элементы управления используют метку соответствующего элемента управления. Элементы управления "Счетчик" всегда являются подчиненными элементами управления.
    -   **Для всех элементов управления выберите самый безопасный (чтобы предотвратить потери данных или доступ к системе), по умолчанию — наиболее безопасное значение.** Если безопасность и безопасность не являются факторами, выберите наиболее вероятное или удобное значение.
    -   **Предпочитать ограниченные элементы управления.** Используйте ограниченные элементы управления, такие как списки и ползунки, когда это возможно, вместо неограниченных элементов управления, таких как текстовые поля, чтобы уменьшить потребность в вводе текста.
    -   **Рекомендуется рассмотреть отключенные элементы управления.** Отключенные элементы управления могут быть трудно использовать, так как пользователям буквально приходится выдавать причины отключения. Отключите элемент управления, когда пользователи хотят применить его, и можно легко определить причину отключения элемента управления. Удалите элемент управления, когда пользователи не смогут его включить или он не должен применять его, или оставьте его включенным, но при неправильном использовании выдает сообщение об ошибке.
        -   **Совет.** Если вы не уверены, следует ли отключить элемент управления или предоставить сообщение об ошибке, начните с создания сообщения об ошибке, которое может быть предоставлено. Если сообщение об ошибке содержит полезную информацию, которую пользователи не могут быстро вывести, оставьте элемент управления включенным и предоставьте ошибку. В противном случае отключите элемент управления.
-   **Кнопки команд**
    -   **Предпочитать конкретные метки по универсальным.** В идеале пользователям не нужно читать что-нибудь еще, чтобы понять метку. Пользователи гораздо чаще читают метки кнопок команд, чем статический текст.
        -   **Исключение:** Не переименовывайте кнопку Cancel, если значение Cancel является неоднозначным. Пользователям не нужно читать все кнопки, чтобы определить, какая кнопка отменяет действие. Однако переименуйте Cancel (Отмена), если неясно, какое действие отменяется, например при наличии нескольких ожидающих действий.
    -   **При запросе вопроса используйте метки, которые соответствуют этому вопросу.** Например, укажите Yes и без ответа на вопрос да или нет.
    -   **Не используйте кнопки Применить в диалоговых окнах, которые не являются страницами свойств или элементами панели управления.** Кнопка «применить» означает применение ожидающих изменений, но оставляет окно открытым. Это позволит пользователям оценивать изменения перед закрытием окна. Однако это требуется только для страниц свойств и элементов панели управления.
    -   Пометка кнопки Cancel (Отмена), если отмена возвращает среду в предыдущее состояние (без побочных эффектов); в противном случае пометка кнопки "Закрыть" (если операция завершена) или "остановлена" (если операция выполняется), чтобы указать, что она оставляет текущее измененное состояние неизменным.
-   **Ссылки на команды**
    -   **Всегда представим ссылки на команды в наборе из двух или более.** Логически нет причин задавать вопрос, у которого есть только один ответ.
    -   **Укажите явную кнопку отмены.** Не используйте для этой цели ссылку на команду. Очень часто пользователи понимают, что они не хотят выполнять задачу. Используя командную ссылку для отмены, пользователи должны будут внимательно считывать все ссылки на команды, чтобы определить, какое из них означает Отмена. Наличие явной кнопки Cancel позволяет пользователям эффективно отменять задачу.
    -   **Если при предоставлении явной кнопки отмены остается единственная ссылка на команду, укажите как команду для отмены, так и кнопку отмены.** В этом случае пользователи станут ясно, что у пользователей есть выбор. Заменяйте эту ссылку на эту команду с точки зрения того, как она отличается от первого ответа, вместо просто "Отмена" или некоторых вариантов.
-   **Флажки "больше не показывать \<item\> "**
    -   **Рассмотрите возможность использования параметра "больше не показывать \<item\> ", чтобы разрешить пользователям подавлять повторяющиеся диалоговые окна, только если нет лучшей альтернативы.** Лучше всегда показывать диалоговое окно, если это действительно требуется пользователю, или просто удалить его, если это не так.
    -   **Замените \<item\> на конкретный элемент.** Например, больше не показывать это напоминание. При ссылке на диалоговое окно в общем случае используйте больше не показывать это сообщение.
    -   **Явно укажите, когда вводимые пользователем данные будут использоваться для будущих значений по умолчанию** , добавив следующее предложение в параметре. по умолчанию в будущем будут использоваться следующие параметры.
    -   **Не выбирайте параметр по умолчанию. Если диалоговое окно действительно должно отображаться только один раз, сделайте это без запроса.** Не используйте этот параметр в качестве оправдания для пользователей докучайте — убедитесь, что поведение по умолчанию не имеет смысла.
    -   **Если пользователь выберет параметр и нажмет кнопку Отмена, этот параметр вступит в силу.** Этот параметр является мета-параметром, поэтому он не соответствует стандартному поведению отмены без побочных эффектов. Обратите внимание, что если пользователи не хотят видеть диалоговое окно в будущем, скорее всего, они также хотят отменить его.
-   **Ссылки**
    -   **Не присваивайте** [ключ доступа](glossary.md). Доступ к ссылкам осуществляется с помощью клавиши TAB.
    -   **Не добавляйте к тексту ссылки "щелкните" или "щелкните здесь".** Это не обязательно, поскольку ссылка подразумевает нажатие кнопки.
-   **Подсказки**
    -   **Используйте подсказки для создания меток для элементов управления без метки.** Для обеспечения согласованности не нужно давать подсказок элементов управления с метками просто.
    -   **Подсказки также могут предоставлять более подробные сведения для кнопок с метками на панели инструментов, если это удобно.** Не просто Повторяйте и не выводите слова, которые уже находятся в метке.
    -   **Избегайте покрытия объекта, с которым пользователь собирается просматривать или взаимодействовать с.** Всегда размещайте TIP на стороне объекта, даже если это требует разделения между указателем и подсказкой. Некоторое разделение не является проблемой, если связь между объектом и его Советом понятна.
        -   **Исключение:** Подсказки полного имени, используемые в списках и деревьях.
    -   **Для коллекций элементов Избегайте появления следующего объекта, с которым пользователь скорее всего будет просматривать или взаимодействовать.** Для горизонтально упорядоченных элементов старайтесь не располагать советы справа; для вертикально упорядоченных элементов старайтесь не размещать советы ниже.
-   **Поэтапное представление информации**
    -   **Чтобы скрыть расширенные или редко используемые параметры, команды и сведения, которые обычно не нужны пользователям, используйте более или менее прогрессивных кнопок раскрытия.** Не скрывать часто используемые элементы, так как пользователи могут их не найти. Но убедитесь, что все скрытые значения имеют значение.
    -   Если на поверхности всегда отображаются некоторые параметры, команды или сведения, используйте следующие пары меток:
        -   **Больше или меньше параметров.** Используйте для параметров или сочетание параметров, команд и сведений.
        -   **Больше или меньше команд.** Используйте только для команд.
        -   **Больше или меньше сведений.** Используйте только для информации.
        -   **Больше или меньше \<object name\> .** Используется для других типов объектов, таких как папки.
    -   В противном случае:
        -   **Показать/скрыть параметры.** Используйте для параметров или сочетание параметров, команд и сведений.
        -   **Показать/скрыть команды.** Используйте только для команд.
        -   **Показать/скрыть сведения.** Используйте только для информации.
        -   **Показать/скрыть \<object name\> .** Используется для других типов объектов, таких как папки.
-   **Индикаторы выполнения**
    -   **Используйте индикаторы хода выполнения для операций, требующих ограниченного количества времени,** даже если это время не может быть точно предсказуемым. Неопределенные индикаторы выполнения показывают, что выполняется, но не предоставляет никакой другой информации. Не выбирайте неопределенный индикатор выполнения, основанный только на возможном отсутствии точности.
    -   **Укажите время, оставшееся для оценки, если это можно сделать точно.** Точность оставшихся оценочных оценок удобна, но оценки, которые отменяют отметку или возобновляются, значительно не полезны. Перед предоставлением точных оценок может потребоваться выполнить некоторую обработку. В этом случае не следует отображать потенциально неточные оценки за этот начальный период.
    -   **Не перезапускайте ход выполнения.** Индикатор выполнения теряет свое значение, если он перезапускается (возможно, из-за завершения операции), так как пользователи не могут узнать, когда операция будет завершена. Вместо этого все шаги, выполняемые в операции, совместно используют часть хода выполнения, и индикатор выполнения переходит в состояние завершения один раз.
    -   **Укажите полезные сведения о ходе выполнения.** Предоставление дополнительных сведений о ходе выполнения, но только в том случае, если пользователи могут выполнять с ними какие-либо действия. Убедитесь, что текст отображается достаточно долго, чтобы пользователи могли прочитать его.
    -   **Не объединяйте индикатор выполнения со занятым указателем.** Используйте один или другой, но не оба одновременно.
-   **Запросы**
    -   **Используйте запрос, если пространство на экране имеет такое вознаграждение, что использование метки или инструкции нежелательно,** например на панели инструментов.
    -   **Запрос в первую очередь определяет назначение текстового поля или поля со списком компактным способом.** Он не должен быть важной информацией, которую пользователь должен видеть при использовании элемента управления.
    -   **Текст подсказки не следует путать с реальным текстом.** Для этого выполните следующие действия.
        -   Нарисуйте текст подсказки курсивом серым цветом, а фактическим текстовым текстом — черным.
        -   Текст подсказки не должен быть редактируемым и должен исчезнуть после нажатия пользователем кнопки или клавиши TAB в текстовом поле.
            -   **Исключение:** Если текстовое поле имеет фокус ввода по умолчанию, отображается запрос, и он исчезает после начала ввода пользователем.
    -   Не используйте закрывающие знаки препинания или многоточие.
-   **Уведомления**
    -   **Используйте уведомления для событий, не связанных с текущими действиями пользователя, не требуя немедленного вмешательства пользователя, и пользователи могут без проблем проигнорировать.**
    -   **Не позлоупотреблять уведомлениями:**
        -   **Используйте уведомления только при необходимости.** При отображении уведомления пользователи могут прерывать работу пользователей или даже раздражать их. Убедитесь, что прерывания выравниваются по ширине.
        -   **Используйте уведомления для некритических событий или ситуаций, не требующих немедленного вмешательства пользователя.** Для критических событий или ситуаций, требующих немедленного вмешательства пользователя, используйте альтернативный элемент пользовательского интерфейса (например, модальное диалоговое окно).
        -   **Не используйте уведомления для объявлений функций!**

## <a name="keyboard"></a>Клавиатура

-   **Назначайте начальные входные данные элементу управления, с которым пользователи с наибольшей вероятностью взаимодействуют первыми,** что часто является первым интерактивным элементом управления. Если первое интерактивное управление не является хорошим выбором, рассмотрите возможность изменения макета окна.
-   **Присвойте вкладкам значения табуляции для всех интерактивных элементов управления, включая поля редактирования только для чтения. Отличи**
    -   Групповые наборы связанных элементов управления, которые ведут себя как один элемент управления, например переключатели. Такие группы имеют одну позицию табуляции.
    -   Правильно содержат группы, чтобы клавиши со стрелками циклически переходили в пределах группы и остаться в группе.
-   **Порядок табуляции должен передаваться слева направо, сверху вниз.** Как правило, порядок табуляции должен следовать за чтением. Рассмотрите возможность создания исключений для часто используемых элементов управления, поместив их раньше в последовательность табуляции. На вкладке должны быть циклически перебираться все позиции табуляции в обоих направлениях без остановки. В пределах группы последовательность табуляции должна быть в последовательном порядке без исключений.
-   **В пределах табуляции порядок клавиш со стрелками должен передаваться слева направо, сверху вниз,** без исключений. Клавиши со стрелками должны циклически проходить по всем элементам в обоих направлениях без остановки.
-   **Представьте кнопки фиксации в следующем порядке:**
    -   Ладно, \[ \] /ЕС
    -   \[Не \] /но
    -   Отменить
    -   Применить (при наличии)

где \[ это \] и не \[ делать, это не \] относится к конкретным ответам на основную инструкцию.

-   **Не путайте ключи доступа с сочетаниями клавиш.** Хотя ключи доступа и сочетания клавиш обеспечивают доступ с клавиатуры к пользовательскому интерфейсу, они имеют разные цели и рекомендации.
-   **Везде, где это возможно, следует назначать уникальные ключи доступа всем интерактивным элементам управления или их меткам.** [Текстовые поля только для чтения](ctrl-text-boxes.md) — это интерактивные элементы управления (так как пользователи могут прокручивать их и копировать текст), поэтому они пользуются преимуществами клавиш доступа. **Не присваивайте ключи доступа:**
    -   Кнопки ОК, отмена и закрыть. Для ключей доступа используются клавиши ВВОД и ESC. Однако всегда назначайте ключ доступа для элемента управления, который означает ОК или Отмена, но имеет другую метку.
-   **Назначьте сочетания клавиш наиболее часто используемым командам.** Нечасто используемые программы и компоненты не нуждаются в сочетаниях клавиш, так как вместо этого пользователи могут использовать ключи доступа.
-   **Не делайте сочетание клавиш единственным способом выполнения задачи.** Пользователи также должны иметь возможность использовать мышь или клавиатуру со вкладками, стрелками и клавишами доступа.
-   **Не назначайте разные значения для хорошо известных сочетаний клавиш.** Так как они зафиксированы, неоднородное значение для хорошо известных сочетаний клавиш — беспроблемные и подверженные ошибкам.
-   **Не пытайтесь назначить сочетания клавиш для программы на уровне системы.** Сочетания клавиш программы вступят в действие только в том случае, если программа имеет фокус ввода.

## <a name="mouse"></a>Мышь

-   **Никогда не нужно, чтобы пользователи щелкнули объект, чтобы определить, можно ли его щелкнуть.** Пользователи должны иметь возможность определять возможности щелчка только в визуальной проверке.
    -   Основной пользовательский интерфейс (например, кнопки фиксации) должен иметь статическое выделение щелчка. Пользователям не нужно навести указатель на обнаружение основного пользовательского интерфейса.
    -   Дополнительный пользовательский интерфейс (например, вторичные команды или элементы управления последовательной раскрытия) может отображать их возможность щелчка при наведении указателя мыши.
    -   Текстовые ссылки должны статически предлагать текст ссылки, а затем отображать свое выделение щелчка (подчеркнутое или другое изменение презентации с помощью указателя руки) при наведении указателя мыши.
    -   Графические ссылки показывают только указатель мыши.
-   **Используйте указатель «рука» (или «ссылка SELECT») только для текстовых и графических ссылок.** В противном случае пользователям придется щелкнуть объекты, чтобы определить, являются ли они ссылками.

## <a name="dialog-boxes"></a>Диалоговые окна

-   **Для модальных диалоговых окон требуется взаимодействие, поэтому используйте их для тех вещей, на которые пользователи должны реагировать перед продолжением задачи.** Убедитесь, что прерывание выровнено, например, для критически важных или редких, одноразовых задач, требующих завершения. В противном случае рассмотрите возможность немодальных вариантов.
-   **Немодальные диалоговые окна не требуют взаимодействия, поэтому их следует использовать, когда пользователям нужно переключаться между диалоговым окном и окном-владельцем.** Они лучше использовать для частых, повторяющихся или текущих задач. Однако ленты, панели инструментов и окна палитры часто являются лучшими альтернативами.

## <a name="property-sheets"></a>Страницы свойств

-   **Убедитесь, что свойства необходимы.** Не следует нагружать страницы свойств с ненужными свойствами, чтобы избежать сложностей при проектировании.
-   **Представление свойств с точки зрения пользователей вместо технологий.** Так как свойство настраивает определенную технологию, это не означает, что необходимо предоставить свойство в терминах этой технологии.
    -   Если необходимо предоставить параметры в терминах технологии (например, если пользователи распознают название технологии), Включите краткое описание преимуществ для пользователей.
-   **Используйте определенные значимые метки вкладки.** Не используйте универсальные метки вкладок, которые могут применяться к любой вкладке, например Общие, расширенные или параметры.
-   **Избегайте использования общих страниц.** Вам не обязательно иметь общую страницу. Используйте страницу «Общие» только в том случае, если:
    -   Свойства применяются к нескольким задачам и являются значимыми для большинства пользователей. Не размещайте специализированные или дополнительные свойства на странице Общие, но вы можете сделать их доступными с помощью кнопки на странице Общие.
    -   Свойства не соответствуют более конкретным категориям. Если это так, используйте вместо этого имя страницы.
-   **Избегайте использования дополнительных страниц.** Используйте страницу Дополнительно, только если:
    -   Свойства применяются к нетипичным задачам и в основном важны для опытных пользователей.
    -   Свойства не соответствуют более конкретным категориям. Если это так, используйте вместо этого имя страницы.
-   **Не используйте вкладки, если окно свойств имеет только одну вкладку и не является расширяемым.** Вместо этого используйте обычное диалоговое окно с кнопками ОК, отмена и необязательное применение. Однако расширяемые окна свойств (которые могут быть расширены третьими сторонами) должны использовать вкладки.

## <a name="wizards"></a>Мастера

-   **Сначала рассмотрите упрощенные варианты, например диалоговые окна, области задач или отдельные страницы.** Мастера являются большим пользовательским интерфейсом, который лучше всего подходит для выполнения многошаговых и редко выполняемых задач. Не нужно использовать мастера — вы можете предоставить полезную информацию и помощь в любом пользовательском интерфейсе.
-   **Используйте Next только при переходе на следующую страницу без обязательств.** Переход на следующую страницу считается обязательством, когда его результат не может быть отменен нажатием кнопки "назад" или "Отмена".
-   **Используйте только для исправления ошибок.** Помимо исправления ошибок, пользователям не придется щелкать кнопку назад, чтобы сделать ход выполнения задачи.
-   **Когда пользователи фиксируются в задаче, используйте кнопку фиксации, которая представляет собой конкретный ответ на основную инструкцию (например, Print, Connect или Start).** Не используйте универсальные метки, такие как Next (что не подразумевает обязательство) или завершения (что не относится к конкретному) для фиксации задачи. Метки на этих кнопках фиксации должны иметь смысл самостоятельно. Всегда запускать метки кнопок фиксации с помощью глагола. **Отличи**
    -   Используйте Finish, если конкретные ответы по-прежнему являются универсальными, например сохранить, выбрать, выбрать или получить.
    -   Используйте кнопку "Готово", чтобы изменить конкретный параметр или коллекцию параметров.
-   **Используйте ссылки на команды только для выбора, а не для обязательств.** Отдельные кнопки фиксации указывают на то, что в мастере гораздо лучше, чем ссылки на команды.
-   **При использовании ссылок на команды скройте кнопку Далее, но оставьте кнопку Отмена.**
-   **Используйте Close для страниц Follow-Up и завершения.** Не используйте Cancel, поскольку закрытие окна не отменяет все изменения или действия, выполненные на этом этапе. Не используйте действие Done, поскольку это не обязательная команда.
-   **Не используйте "Wizard" в именах мастеров.** Например, используйте "подключиться к сети", а не "Мастер настройки сети". Однако в качестве мастеров можно ссылаться на мастера. Например: "Если вы настраиваете сеть в первый раз, можете получить помощь с помощью мастера подключения к сети".
-   **Сохранение выбора пользователей с помощью навигации.** Например, если пользователь вносит изменения, нажимает кнопку «назад», затем «Далее» эти изменения должны сохраняться. Пользователи не хотят повторно вводить изменения, если они явно не решили их очистить.

## <a name="wizard-pages"></a>Страницы мастера

-   **Сосредоточьтесь на принятии эффективных решений.** Сократите количество страниц, чтобы сосредоточиться на Essentials. Консолидируйте связанные страницы и выдавайте необязательные страницы из основного потока. При первом нажатии кнопки "Далее" в мастере может показаться хорошей опыт, но если пользователям не нужно изменять значения по умолчанию, страницы, вероятно, не нужны.
-   **Не используйте страницы приветствия — сделайте первую страницу функциональной, когда это возможно.** Используйте необязательную начало работыную страницу только в следующих случаях:
    -   Мастер имеет необходимые компоненты, необходимые для успешного завершения работы мастера.
    -   Пользователи могут не понимать назначение мастера на основе первой страницы выбора, и нет места для дальнейшего объяснения.
    -   Главная Инструкция для начало работы страниц — это "перед началом:".
-   **На страницах, в которых пользователи просят выбрать вариант, оптимизируются для наиболее вероятных случаев.** Эти виды страниц должны представлять фактические параметры, а не только инструкции.
    -   Если вы не используете начало работы страницу, объясните назначение мастера в верхней части первой страницы вариантов.
-   **Используйте страницы фиксации, чтобы сделать ее ясной, когда пользователи фиксируются в задаче.** Как правило, страница фиксации является последней страницей выбора, а кнопка Далее перемечается для указания на то, что задача зафиксирована.
    -   Не используйте страницы сводки, которые просто суммируют предыдущее значение пользователя, если только задача не является опасной (включая безопасность или потери времени или денег), или существует хорошая вероятность того, что пользователи не смогут понять их параметры и их необходимо будет проанализировать.
-   **Не используйте страницы с поздравлениями, которые не выполняют никаких действий, но завершают работу мастера.** Если результаты мастера очевидны для пользователей, просто закройте мастер на последней кнопке фиксации.
    -   Используйте Follow-Up страницы при наличии связанных задач, которые пользователи, скорее всего, будут выполнять в качестве дальнейших действий. Старайтесь не выполнять привычные задачи, такие как «отправка сообщения электронной почты».
    -   Используйте страницы завершения только в том случае, если результаты не видны и нет лучшего способа предоставить отзыв о завершении задачи.
    -   Для указания завершения задач в мастерах со страницами хода выполнения должна использоваться страница завершения или страница Follow-Up. Для выполнения длительных задач закройте мастер на странице фиксации и используйте уведомления, чтобы отправить отзыв.

## <a name="error-messages"></a>Сообщения об ошибках

-   **Не выдавать сообщения об ошибках, когда пользователи, скорее всего, не будут выполнять действие или изменить их поведение** в результате сообщения. Если нет действий, которые могут предпринять пользователи, или если проблема не существенна, подавление сообщения об ошибке.
-   **По возможности предлагать решение, чтобы пользователи могли решить проблему.** Однако убедитесь, что предлагаемое решение может решить проблему. Не изменяйте время пользователей, предлагая возможные, но невероятные решения.
-   **Быть конкретным.** Старайтесь не использовать неясные слова, такие как синтаксическая ошибка и Недопустимая операция. Укажите конкретные имена, расположения и значения задействованных объектов.
-   **Не используйте выражения, которые бламес пользователя или подразумевают ошибку пользователя.** Старайтесь не использовать вас и в выражениях. В то время как активная речь является предпочтительной, используйте пассивный язык, когда пользователь является субъектом, и может наизмененся на ошибку, если была использована активная Voice.
-   **Не используйте "ОК" для сообщений об ошибках.** Пользователи не будут видеть ошибки как есть. Если сообщение об ошибке не имеет прямого действия, используйте вместо него Close.
-   **Не используйте следующие слова:**
    -   Ошибка, сбой (используйте вместо него проблему)
    -   Не удалось (используйте вместо этого)
    -   Недопустимое, недопустимое, плохое (используйте неверное или недопустимое)
    -   Abort, Kill и Terminate (используйте вместо этого использование команды "остановить")
    -   Разрушительные, неустранимые (используйте серьезную серьезность)

Эти термины не требуются и не имеют сигнала о поощрении Windows. Вместо этого [при правильном использовании](vis-std-icons.md)значка ошибки в достаточной степени сообщается о проблеме.

-   **Не налагаются сообщения об ошибках с звуковыми эффектами.** Это жарринг и не требуется.

## <a name="warning-messages"></a>Предупреждающие сообщения

-   **Предупреждения описывают условие, которое может привести к возникновению проблемы в будущем.** Предупреждения не являются ошибками или вопросами, поэтому не следует заменять стандартные вопросы как предупреждения.
-   **Не выдавать предупреждающие сообщения, когда пользователи, скорее всего, не будут выполнять действие или изменить их поведение в результате сообщения.** Если нет действий, которые могут предпринять пользователи, или если условие не имеет значения, подавляет предупреждающее сообщение.

## <a name="confirmations"></a>Подтверждения

-   **Не используйте ненужные подтверждения.** Используйте только подтверждения, если:
    -   По ясной причине не следует продолжать работу и разумная вероятность того, что иногда они не будут.
    -   Действие имеет значительные последствия или не может быть просто отменено.
    -   Действие имеет последствия, о которых пользователи могут не знать.
    -   Чтобы продолжить выполнение действия, необходимо, чтобы пользователи выбрали вариант, не имеющий подходящего значения по умолчанию.
    -   Учитывая текущий контекст, пользователи, скорее всего, выполнили действие с ошибкой.
-   **Подтверждать фразы как «да» или «нет» и предоставлять ответ «да» или «нет».** В отличие от других типов диалоговых окон, подтверждения предназначены для того, чтобы пользователи не могли принимать быстрые решения. Если пользователи не подумали о своем ответе, подтверждение не имеет значения.

## <a name="icons"></a>Значки

-   **Все значки должны соответствовать** [рекомендациям по значку в стиле Aero](vis-icons.md). Замените все значки в стиле Windows XP.
-   **Выберите стандартные значки на основе их типа сообщений, а не важность базовой проблемы:**
    -   **План.** Произошла ошибка или возникшая проблема.
    -   **!.** Условие, которое может вызвать проблему в будущем.
    -   **Об.** Полезная информация.

Если проблемы объединяются в разных типах сообщений, сосредоточьтесь на наиболее важных аспектах, с которыми должны действовать пользователи.

-   **Значки должны всегда соответствовать основной инструкции или другому соответствующему тексту.**
-   **Значки ошибок не требуются для некритических проблем ввода данных пользователем.** Тем не менее для ошибок на месте требуются значки, так как в противном случае такие контекстные Отзывы можно было бы не заметить.
-   **Не используйте значки предупреждений для «смягчения» некритических ошибок.** Ошибки не являются предупреждениями; Вместо этого примените рекомендации значка ошибки.
-   **В диалоговых окнах вопросов используйте значки предупреждений только для вопросов с важными последствиями.** Не используйте значки предупреждений для часто задаваемых вопросов.

## <a name="help"></a>Справка

-   **Ссылка на соответствующие разделы справки.** Не следует ссылаться на домашнюю страницу справки, оглавление, список результатов поиска или страницу, которая просто ссылается на другие страницы. Избегайте связывания со страницами, структурированными в виде большого списка часто задаваемых вопросов, так как она заставляет пользователей искать те, которые соответствуют выбранным ссылкам. Не применяйте ссылки на конкретные разделы справки, которые не имеют значения и могут оказаться полезными для поставленной задачи. Никогда не ссылаться на пустые страницы.
-   **Не намещайте ссылки справки на каждое окно или страницу для обеспечения согласованности.** Предоставление ссылки на справку в одном месте не означает, что их необходимо предоставить везде.
-   **По возможности текст справки по фразам содержит ссылки на основной вопрос, на который вы ответили в содержимом справки.** Не используйте выражения "Дополнительные сведения о программе" или "получите помощь в этом".
-   **Используйте ссылку "вся Справка" для текста ссылки,** а не только для ключевых слов.
-   **Используйте полные предложения.**
-   **Не используйте закрывающие знаки препинания и многоточия, кроме вопросительных знаков.**
-   **Если содержимое справки находится в оперативном режиме, сделайте его открытым в тексте ссылки.** Это позволит сделать результаты прогнозируемыми.
