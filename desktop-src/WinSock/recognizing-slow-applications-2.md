# <a name="recognizing-slow-applications"></a>Распознавание замедляют работу приложений

Это пошаговое руководством определяет *медленную работу* приложения Microsoft Windows с непарной производительностью. В медленных приложениях имеется одна или несколько следующих симптомов:

-   Использование ЦП и сети мало.

    Компьютер ожидает чего-либо. Часто приложение ожидает в сети.

-   Отключение алгоритма Nagle с помощью \_ параметра сокета TCP-задержки повышает производительность.

    Это указывает на другие проблемы и не должно рассматриваться как решение. Отключение алгоритма Nagle увеличивает нагрузку на протокол. Не используйте этот метод в качестве исправления для неработающих приложений — только в качестве указания, что приложению требуется другая работа для устранения проблем с производительностью.

-   Приложение имеет высокую нагрузку.

    Чтобы вычислить нагрузку на приложения, определите объем данных, предназначенных для перемещения в каждом направлении. Затем используйте netstat и добавьте (для Ethernet) 60 байт для каждого пакета и 500 байт для каждого подключения. Для потоковой передачи по сети Ethernet лучше всего около 6% накладных расходов. При подключении через модем лучшие накладные расходы повышены примерно на 2% из-за того, что канал PPP использует сжатие заголовков. Дополнительные сведения см. [в разделе Вычисление издержек с помощью netstat](calculating-overhead-with-netstat-2.md) .

-   Отклик приложения замедлится, когда соединение имеет большой RTT.

    Если приложение не приближается к пропускной способности связи, большое время приема-передачи не должно оказывать никакого влияния. Значительное снижение производительности при большом RTT — это четкий знак сериализованной обработки и много небольших транзакций.

Каждое приложение должно быть протестировано в среде с большим RTT. Это открывает большинство приложений, которые пострадает от неудовлетворительных вариантов разработки. Это тестирование может выполняться в нескольких средах, включая беспроводную ЛОКАЛЬную сеть, симулятор с задержкой канала или вспомогательную сеть.
