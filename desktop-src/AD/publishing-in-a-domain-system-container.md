# <a name="publishing-in-a-domain-system-container"></a>Публикация в контейнере системы домена

Системный контейнер раздела домена содержит операционные сведения для каждого домена. К ним относятся локальная политика безопасности по умолчанию, отслеживание ссылок на файлы, сетевые собрания и контейнеры для точек подключения Windows Sockets Registration and Resolution (РНР) и службы имен RPC (Рпкнс). Контейнер системы по умолчанию скрыт и предоставляет удобное место для хранения объектов, представляющих интерес для администраторов, но не для конечных пользователей.

Службы, которые не привязаны к одному узлу, могут создавать свои запрашивая в системном контейнере раздела домена. Эта альтернатива может быть полезна для служб с репликами, установленными на нескольких узлах, Каждая реплика предоставляет клиентам одинаковые службы на всем домене. Она позволяет сгруппировать все объекты для реплицированной службы в одном контейнере.

Службы, создающие зависящие от службы объекты в контейнере System, должны выполнять следующие действия:

1.  Создайте вложенный контейнер **контейнера** класса объекта как непосредственный дочерний элемент контейнера System. Присвойте этому подконтейнеру имя, которое четко идентифицирует его как отношение к службе.
2.  Создайте объекты, связанные со службой, в этом подконтейнере. Например, NetMeeting использует контейнер meetings для публикации объектов сетевого собрания.

Поставщик с несколькими продуктами может использовать аналогичную стратегию группирования объектов, связанных со службами, для всех своих продуктов. В этом случае можно создать объект- **контейнер** с именем, которое ясно идентифицирует поставщика. Затем создайте объекты **контейнера** для каждой службы в качестве дочерних элементов контейнера поставщика. Создайте контейнер для конкретного поставщика в качестве дочернего для системного контейнера.
